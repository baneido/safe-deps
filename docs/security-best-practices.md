# Security Best Practices for Package Manager Linting

Last reviewed: 2026-06-20

This document summarizes package-management security practices that are useful
for `safe-deps` lint rules. The focus is static or CI-configurable signals that
can be detected without executing arbitrary project code.

## Scope

Primary targets:

- JavaScript and TypeScript: `npm`, `yarn`, `pnpm`, `bun`
- Python: `pip`, `uv`

Adjacent ecosystems worth supporting after the MVP:

- PHP: Composer
- Ruby: Bundler
- Rust: Cargo
- Go: Go modules
- JVM: Gradle and Maven
- .NET: NuGet

## Baseline Principles

1. Commit a lockfile and install from it in CI.
   Dependency resolution should be reproducible and reviewable. CI should fail
   when the manifest and lockfile disagree.

2. Treat integrity metadata as security-sensitive.
   Lockfile hashes, checksums, registry signatures, and local hash pins should
   not be disabled or silently regenerated in CI.

3. Restrict install-time code execution.
   Lifecycle scripts, build scripts, plugins, native builds, editable installs,
   and VCS dependencies can execute or introduce code outside ordinary package
   resolution. Prefer default-deny or explicit allowlists where the ecosystem
   supports them.

4. Control package origins.
   Use HTTPS registries, avoid disabling TLS verification, and avoid unsafe
   multi-index behavior that enables dependency confusion.

5. Separate resolution, installation, auditing, and publishing policy.
   A linter should not assume that "latest" is safer. It should check whether
   the project has a controlled update path, a vulnerability-audit path, and a
   publish-authentication/provenance path when publishing is in scope.

6. Keep project and system environments separate.
   CI and development installs should use isolated project environments rather
   than mutating global or externally managed language installations.

## Common Rule Categories

### Reproducible Installs

Detect:

- Manifest with dependencies but no ecosystem lockfile.
- CI using a resolving install command instead of a frozen/locked command.
- Config that prevents lockfile use or lockfile creation.
- Lockfile format known to be obsolete or generated by a different package
  manager without an explicit migration plan.

Examples:

- `npm ci` instead of `npm install` in CI.
- `yarn install --immutable`; add `--immutable-cache --check-cache` for
  zero-install projects accepting external pull requests.
- `pnpm install --frozen-lockfile` or a CI environment where that default is
  guaranteed.
- `bun ci` or `bun install --frozen-lockfile`.
- `uv sync --locked` or `uv lock --check` in CI.
- `pip install --require-hashes -r requirements.txt` for deployment-style
  requirements files.

### Integrity and Signature Validation

Detect:

- Disabled checksum/hash behavior.
- Commands that update checksums in normal CI paths.
- Missing hashes in deployment requirements.
- Signature verification available but absent in higher-assurance profiles.

Examples:

- npm lockfiles include package integrity metadata; `npm audit signatures`
  verifies registry signatures and provenance attestations when supported.
- Yarn's `checksumBehavior` should not be `ignore`, and `update` should not be
  accepted in CI without an explicit review process.
- pnpm treats tarball integrity mismatch as a hard failure by default; flag
  `--update-checksums` in CI as suspicious.
- pip's secure mode requires `--require-hashes` and pinned requirements.

### Install-Time Scripts and Native Builds

Detect:

- Lifecycle scripts enabled globally without allowlisting.
- `ignore-scripts=false` in high-risk CI contexts.
- pnpm build-script allowlist bypasses such as `dangerouslyAllowAllBuilds`.
- Bun `trustedDependencies` entries that are broad, unnecessary, or include
  non-registry packages without review.
- Composer plugins or scripts allowed without explicit configuration.

Guidance:

- Default-deny script execution is strongest where available.
- When packages need build scripts, require a narrow allowlist and keep it in
  version control.
- Flag changes to a dependency's build scripts for review.

### Registry and Index Safety

Detect:

- HTTP package registries or explicit TLS verification bypasses.
- npm/Yarn/pnpm/Bun registry URLs using plaintext HTTP outside local test
  exceptions.
- pip `--trusted-host`, `PIP_TRUSTED_HOST`, or `--extra-index-url` without an
  explicit dependency-confusion mitigation.
- uv `index-strategy = "unsafe-best-match"` or `allow-insecure-host`.
- Broad Git dependency allowlists.

Guidance:

- Prefer one trusted source per package namespace.
- For private packages, scope or explicitly bind packages to private indexes.
- Avoid mixing private and public indexes with "best match" resolution unless
  the organization has a separate control preventing name collision.

### Vulnerability and Malware Checks

Detect:

- Audit command intentionally disabled.
- No audit workflow in CI for projects with external dependencies.
- Ignore lists without comments, expiry, or advisory identifiers.
- Malware checks supported but disabled in higher-assurance profiles.

Examples:

- `npm audit`, `yarn npm audit -A -R`, `pnpm audit`, and `bun audit`.
- Python projects can use `pip-audit`; uv also has preview on-sync malware
  checking via OSV.
- Go projects should run `govulncheck`.
- Composer has `composer audit`.

Severity should depend on project policy. Missing audit automation is usually a
warning, not a hard error, unless the repo declares a strict security profile.

### Dependency Source Hygiene

Detect:

- Unpinned Git branch dependencies, such as `main`, `master`, or floating tags.
- Direct tarball URLs without integrity metadata.
- Local path dependencies in production/deployment manifests.
- VCS dependencies over SSH where CI provenance is hard to verify.
- Deprecated package-source mechanisms such as direct `setup.py install`.

Guidance:

- Pin VCS dependencies to immutable commits when unavoidable.
- Prefer registry packages with lockfile integrity data.
- Require explicit justification for path, link, file, git, github, or tarball
  dependencies in production dependency sets.

### Publishing and Provenance

Detect:

- Publishing workflows using long-lived tokens where trusted publishing/OIDC is
  supported.
- Missing provenance generation for packages that are published from CI.
- npm publishing without 2FA or trusted publishing policy signals.
- uv publishing with trusted publishing disabled where a supported CI
  environment is used.

This is secondary for an install-focused linter, but it is important for
repositories that publish packages consumed by others.

## Ecosystem Notes

### npm

Primary files:

- `package.json`
- `package-lock.json` or `npm-shrinkwrap.json`
- `.npmrc`
- CI workflow files

Recommended checks:

- Require a committed lockfile for applications and services.
- Prefer `npm ci` in CI; it requires a lockfile, fails when the lockfile and
  manifest disagree, removes existing `node_modules`, and does not rewrite
  manifests or lockfiles.
- Flag `.npmrc` values such as `package-lock=false`, `strict-ssl=false`, or
  plaintext `registry=http://...`.
- Flag `--force`, `--legacy-peer-deps`, and `--ignore-scripts=false` in strict
  CI profiles when they are used to bypass normal install safeguards.
- Encourage `npm audit` or equivalent vulnerability scanning.
- For high-assurance profiles, check for `npm audit signatures`.
- For packages that publish from CI, check for trusted publishing or provenance
  configuration.

Risk-specific notes:

- `npm install` mutates the lockfile by design. It is suitable for development
  and dependency updates, not for reproducible CI installs.
- `npm audit fix` runs an install under the hood and can change the dependency
  tree. Treat it as an update operation, not a passive scan.

### Yarn

Primary files:

- `package.json`
- `yarn.lock`
- `.yarnrc.yml`
- `.pnp.cjs` and `.yarn/cache` for zero-install projects

Recommended checks:

- Prefer `yarn install --immutable` in CI.
- For zero-install projects that commit cache artifacts and accept external
  pull requests, prefer `--immutable-cache --check-cache`.
- Flag `checksumBehavior: ignore`; treat `checksumBehavior: update` in CI as
  review-required.
- Prefer `enableHardenedMode: true` for public pull-request validation, or
  verify that Yarn's environment-triggered hardened mode covers the workflow.
- Check `enableScripts`. Yarn 4.14+ does not run postinstalls by default, but
  projects can re-enable them globally or per package.
- Consider `npmMinimalAgeGate` as a supply-chain hardening rule.
- Flag non-empty `unsafeHttpWhitelist` outside local/test exceptions.
- Use `approvedGitRepositories` where Git dependencies are allowed.
- Use `yarn npm audit -A -R` for whole-project audits in workspaces.

Risk-specific notes:

- Yarn v1 and Yarn Berry have different configuration models. The linter should
  detect the version before applying Yarn-specific rules.

### pnpm

Primary files:

- `package.json`
- `pnpm-lock.yaml`
- `pnpm-workspace.yaml`
- `.npmrc`

Recommended checks:

- Require `pnpm-lock.yaml` for applications and workspaces.
- Prefer `pnpm install --frozen-lockfile` in CI. pnpm defaults this to true in
  many CI environments when a lockfile exists, but explicit commands are easier
  to audit.
- Flag `--no-lockfile`.
- Flag `--update-checksums` in normal CI installs; it is a deliberate integrity
  bypass/update path and should be review-only.
- Keep `strictDepBuilds: true` and configure `allowBuilds` for packages that
  are permitted to run build scripts.
- Flag `dangerouslyAllowAllBuilds`.
- Consider `minimumReleaseAge` to reduce risk from newly published malicious
  packages.
- Run `pnpm audit`; for signature validation, use `pnpm audit signatures` where
  registries publish ECDSA signing keys.

Risk-specific notes:

- pnpm's strict `node_modules` model reduces phantom-dependency risk compared
  with hoisted installs. Flag `shamefully-hoist` as compatibility-risk/security
  warning rather than as a hard error.

### Bun

Primary files:

- `package.json`
- `bun.lock`
- `bunfig.toml`
- `.npmrc`

Recommended checks:

- Require committed `bun.lock`.
- Prefer `bun ci` or `bun install --frozen-lockfile` in CI.
- Flag `BUN_CONFIG_SKIP_SAVE_LOCKFILE` and `BUN_CONFIG_SKIP_LOAD_LOCKFILE` in
  CI.
- Keep lifecycle script execution restricted. Bun defaults to not executing
  arbitrary dependency lifecycle scripts; `trustedDependencies` is the explicit
  allowlist.
- Treat `trustedDependencies` as security-sensitive. `trustedDependencies: []`
  is the strictest mode; explicit entries should be reviewed.
- Prefer `linker = "isolated"` for workspaces and new projects where compatible.
- Consider `minimumReleaseAge` for newly resolved packages.
- Run `bun audit`.

Risk-specific notes:

- Defining `trustedDependencies` replaces Bun's default trusted list rather than
  extending it. A linter should avoid naive "non-empty is good" logic.

### pip

Primary files:

- `requirements*.txt`
- `constraints*.txt`
- `pyproject.toml`
- `pip.conf`, `pip.ini`, environment variables in CI

Recommended checks:

- For deployable applications, require exact pins for both direct and
  transitive dependencies.
- Prefer `--require-hashes` for deployment requirements files. pip requires
  hashes for all requirements and dependencies in hash-checking mode.
- Prefer `--only-binary :all:` in strict deploy profiles to avoid arbitrary
  source-distribution build execution.
- Flag `--trusted-host`, `PIP_TRUSTED_HOST`, HTTP indexes, and TLS bypasses.
- Flag `--extra-index-url` unless private-package names are otherwise protected
  from dependency confusion.
- Flag `--no-build-isolation` and `--break-system-packages` in CI unless the
  project explicitly documents why they are necessary.
- Prefer virtual environments or another project-local environment manager.
- Use `pip check` for installed environment consistency and `pip-audit` for
  known vulnerabilities.

Risk-specific notes:

- pip is not a complete workflow manager. A linter should support both
  requirements-based projects and `pyproject.toml` projects without assuming a
  single canonical lockfile.
- PyPI URL fragments protect against download corruption, but pip's secure
  install guidance distinguishes those remote hashes from local hashes used by
  `--require-hashes`.

### uv

Primary files:

- `pyproject.toml`
- `uv.lock`
- `uv.toml`
- CI workflow files

Recommended checks:

- Require `uv.lock` for uv-managed applications and workspaces.
- In CI, check lockfile freshness with `uv lock --check`, or use commands with
  `--locked`.
- Prefer `uv sync --locked` for reproducible environment creation.
- Avoid `--frozen` when the intended check is "lockfile must match metadata";
  `--frozen` skips that freshness check.
- Keep uv's default `first-index` behavior for multiple indexes. Flag
  `unsafe-best-match` because it exposes dependency-confusion risk.
- Flag `allow-insecure-host`.
- Consider `UV_MALWARE_CHECK=1` in strict profiles while uv's malware checking
  remains preview.
- For publishing workflows, prefer `trusted-publishing = "always"` when running
  in a supported CI environment.

Risk-specific notes:

- uv's `uv.lock` is cross-platform and can represent multiple Python/platform
  marker outcomes. The linter should avoid assuming one resolved artifact per
  package.
- `uv run --no-sync` and inexact sync modes can be useful locally but should be
  scrutinized in CI and release workflows.

### Composer

Primary files:

- `composer.json`
- `composer.lock`
- Composer config

Recommended checks:

- Commit `composer.lock` for applications.
- Use `composer install` in CI/deploy so the lockfile's exact versions are used.
- Run `composer audit` or `composer install --audit`.
- Keep `secure-http` enabled.
- Treat `--no-plugins` and `--no-scripts` as hardening options for untrusted
  installs; otherwise require explicit plugin allowlisting.
- Flag `--ignore-platform-reqs` in CI/deploy.

Risk-specific notes:

- Composer plugins execute PHP code during Composer operations. Flag plugins
  that run without an explicit allowlist.

### Bundler

Primary files:

- `Gemfile`
- `Gemfile.lock`
- `.bundle/config`

Recommended checks:

- Commit `Gemfile.lock` for applications.
- Use deployment/frozen install mode in CI/deploy.
- Enable `lockfile_checksums` for new lockfiles where possible.
- Flag `ssl_verify_mode` values that disable peer verification.
- Audit with an appropriate Ruby advisory tool if project policy requires it.

Risk-specific notes:

- Libraries and applications treat lockfiles differently in Ruby. The linter
  should infer application vs library context before making lockfile absence a
  hard error.

### Cargo

Primary files:

- `Cargo.toml`
- `Cargo.lock`
- `.cargo/config.toml`

Recommended checks:

- Commit `Cargo.lock` for binaries and applications. Libraries may omit it when
  they want downstream users to resolve compatible versions.
- Use `cargo build --locked`, `cargo test --locked`, or equivalent CI commands.
- Flag Git dependencies that are not pinned to immutable revisions.
- Prefer crates.io or approved registries; review source replacement and custom
  registry config.
- Audit with RustSec/cargo-audit or equivalent if project policy requires it.

Risk-specific notes:

- Cargo's lockfile policy is intentionally different for libraries and
  applications. Rule severity must reflect package type.

### Go Modules

Primary files:

- `go.mod`
- `go.sum`

Recommended checks:

- Require `go.sum` for modules with external dependencies.
- Run `go mod verify` in higher-assurance CI.
- Use `-mod=readonly` or CI patterns that fail on implicit `go.mod` changes.
- Run `govulncheck`.
- Configure `GOPRIVATE`, `GONOSUMDB`, and `GONOPROXY` for private modules so
  private paths do not leak to public infrastructure and checksum behavior is
  intentional.
- Flag broad `replace` directives and local filesystem replacements in CI.

Risk-specific notes:

- Go uses a checksum database by default for public modules. Private-module
  configuration is the main static risk surface.

### Gradle and Maven

Primary files:

- Gradle: `build.gradle`, `build.gradle.kts`, `settings.gradle`, dependency
  lockfiles, dependency verification metadata
- Maven: `pom.xml`, wrapper files, repository settings

Recommended checks:

- For Gradle, enable dependency locking where reproducibility is required.
- For Gradle, use dependency verification metadata for checksums/signatures.
- Use repository content filters to bind groups to repositories and reduce
  dependency-confusion risk.
- Prefer HTTPS repositories and flag `allowInsecureProtocol`.
- For Maven, avoid dynamic versions such as `LATEST`, `RELEASE`, and broad
  version ranges in application builds.
- Use Maven/Gradle wrapper with pinned distribution checksums.
- Add vulnerability scanning via organization-approved tooling.

Risk-specific notes:

- Maven does not have a universal lockfile equivalent. Rules should focus on
  fixed versions, repository policy, wrapper integrity, and audit tooling.

### NuGet

Primary files:

- `*.csproj`, `Directory.Packages.props`
- `packages.lock.json`
- `NuGet.config`

Recommended checks:

- Use `packages.lock.json` and locked restore mode for reproducible restores.
- Enable NuGet auditing in CI.
- Prefer central package management for large repos.
- Use trusted package sources and HTTPS feeds.
- Flag broad package source mappings or missing package source mapping in
  private-feed environments.

Risk-specific notes:

- Package source mapping is a strong dependency-confusion mitigation for NuGet
  projects that mix public and private feeds.

## Suggested MVP Rule Set

The first implementation should prioritize high-confidence static checks:

| Rule ID | Category | Severity | Signal |
| --- | --- | --- | --- |
| SD001 | Lockfile missing | error for apps, warning for libraries | Manifest has dependencies but expected lockfile is absent. |
| SD002 | Non-frozen CI install | error | CI uses `npm install`, `yarn install`, `pnpm install`, `bun install`, `uv sync`, or `pip install` without frozen/locked/hash controls. |
| SD003 | TLS/HTTP registry risk | error | Registry/index uses HTTP or disables certificate validation outside local exceptions. |
| SD004 | Integrity disabled | error | Checksum/hash validation is disabled or lockfile use is disabled. |
| SD005 | Script/build bypass | warning or error | Lifecycle/build scripts globally enabled, build allowlists bypassed, or plugin execution unrestricted. |
| SD006 | Unsafe package source | warning | Floating Git branches, direct tarballs, local path deps, or unpinned VCS dependencies in production dependency sets. |
| SD007 | Multi-index dependency confusion | warning or error | pip `--extra-index-url`, uv unsafe index strategy, missing source mapping, or unscoped private registry use. |
| SD008 | Audit missing or disabled | warning | Known package audit command is absent from CI or explicitly disabled. |
| SD009 | Dangerous install flags | warning or error | `--force`, `--ignore-platform-reqs`, `--break-system-packages`, `--no-build-isolation`, `--no-lockfile`, or equivalent. |
| SD010 | Publish provenance absent | info or warning | Publish workflow lacks trusted publishing, provenance, or token-hardening signals. |

## Implementation Notes

- Detect ecosystem version before applying version-specific rules. Yarn v1 vs
  Yarn Berry and old vs new lockfile formats need different checks.
- Separate "application", "library", and "tooling-only" profiles. Lockfile and
  exact-pin expectations differ by profile.
- Do not flag every semver range as insecure. Ranges are normal in libraries;
  frozen lockfiles and CI resolution controls are more important for apps.
- Prefer policy files over command-line heuristics when both exist. Example:
  `.yarnrc.yml`, `pnpm-workspace.yaml`, `bunfig.toml`, `uv.toml`,
  `NuGet.config`, and Gradle verification metadata.
- CI parsing should cover GitHub Actions first, then GitLab CI, CircleCI,
  Buildkite, Jenkins, and generic shell scripts.
- Every suppress/ignore rule should support a reason and optional expiry. This
  is especially important for vulnerability advisory ignores and script
  allowlists.
- Keep networked vulnerability lookup out of the default linter path. The
  default linter can verify that audit tooling is configured; an optional mode
  can execute scanners.
